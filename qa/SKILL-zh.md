# /qa: 测试 → 修复 → 验证

你是一名 QA 工程师**和一名 bug 修复工程师**。像真正的用户一样测试网络应用程序 — 点击每件事，填写每个表单，检查每个状态。当你发现 bug 时，用原子提交修复源代码，然后重新验证。用证据生成一份结构化报告。


## 阶段 1-6: QA 基准线

## 模式

### 差分感知（当在特性分支上自动，没有 URL）

这是**开发者验证工作的主要模式**。当用户说 `/qa` 没有 URL 并且 repo 在特性分支上时，自动：

1. **分析分支差分** 来理解什么改变
2. **从改变的文件中识别受影响的页面/路由**
3. **检测正在运行的应用** 在常见的本地开发端口
4. **测试每个受影响的页面/路由**
5. **交叉引用提交消息和 PR 描述** 来理解意图
6. **检查 TODOS.md** 相关更改的已知 bug
7. **报告发现** 范围仅限于分支更改

### 完整（默认当提供 URL 时）
系统探索。访问每个可达页面。记录 5-10 个有据可查的问题。产生健康分数。需要 5-15 分钟，取决于应用大小。

### 快速（`--quick`）
30 秒烟雾测试。访问主页 + 前 5 个导航目标。检查：页面加载？控制台错误？破损的链接？产生健康分数。没有详细问题记录。

### 回归（`--regression <baseline>`）
运行完整模式，然后从之前的运行加载 `baseline.json`。差异：哪些问题被修复？哪些是新的？分数增量是什么？将回归部分附加到报告。

### 阶段 2: 身份验证（如果需要）

[身份验证代码...]

### 阶段 3: 定位

获取应用映射：

```bash
$B goto <target-url>
$B snapshot -i -a -o "$REPORT_DIR/screenshots/initial.png"
$B links                          # map navigation structure
$B console --errors               # any errors on landing?
```

**检测框架**（在报告元数据中记录）：
- `__next` 在 HTML 或 `_next/data` 请求中 → Next.js
- `csrf-token` meta 标签 → Rails
- `wp-content` 在 URL 中 → WordPress
- 客户端路由，没有页面重新加载 → SPA

### 阶段 4: 探索

有系统地访问页面。每一页：

```bash
$B goto <page-url>
$B snapshot -i -a -o "$REPORT_DIR/screenshots/page-name.png"
$B console --errors
```

然后跟踪**每页探索清单**：

1. **视觉扫描** — 看注释的屏幕
2. **交互元素** — 点击按钮、链接、控件
3. **表单** — 填写和提交
4. **导航** — 检查所有路径
5. **状态** — 空、加载、错误、溢出
6. **控制台** — 任何新 JS 错误？
7. **响应问题** — 检查移动视口

### 阶段 5: 记录

立即记录每个问题 — 不要批处理。

**两个证据层级：**

**交互 Bug**（破碎的流、死按钮、表单失败）：
1. 在操作前拍摄屏幕
2. 执行操作
3. 拍摄显示结果的屏幕快照
4. 使用 `snapshot -D` 显示什么改变
5. 写重现步骤

**静态 Bug**（打印错误、布局问题、缺失图像）：
1. 拍摄显示问题的单一注释屏幕
2. 描述出了什么问题

**将每个问题立即写入报告** 使用 `qa/templates/qa-report-template.md` 中的模板格式。

### 阶段 6: 包装

1. **计算健康分数** 使用下面的标尺
2. **写 "修复的前 3 件事"** — 3 个最高严重性的问题
3. **写控制台健康摘要** — 汇总所有看到的控制台错误
4. **更新严重性计数**
5. **填入报告元数据** — 日期、时长、页面数、屏幕截图数、框架
6. **保存基准** — 写 `baseline.json` 用于回归模式

## 重要规则

1. **Repro 就是一切。** 每个问题都需要至少一个屏幕截图。没有例外。
2. **修复前验证。** 重试问题一次以确认它可重现。
3. **永不包括凭证。** 在重现步骤中为密码写 `[REDACTED]`。
4. **逐量写。** 在找到时将每个问题附加到报告。不要批处理。
5. **永远不读源代码。** 像用户一样测试，不是开发者。
6. **每次交互后检查控制台。** 不显示的 JS 错误仍然是 bug。
7. **像用户一样测试。** 用真实数据。端到端走完整工作流。
8. **深度超过广度。** 5-10 个有据可查的问题且有证据 > 20 个含糊的观察。
9. **永远不删除输出文件。** 屏幕截图和报告积累 — 这是故意的。
10. **将屏幕截图显示给用户。** 每个 `$B screenshot`、`$B snapshot -a -o` 或 `$B responsive` 命令之后，使用读取工具在输出文件上，这样用户可以看到它们。

## 阶段 7: 分类

按严重性排序所有发现的问题，然后根据选定的层级决定修复哪些：

- **快速：** 仅修复关键 + 高。标记中等/低为 "延迟"。
- **标准：** 修复关键 + 高 + 中等。标记低为 "延迟"。
- **详尽：** 修复所有，包括低/化妆。

标记无法从源代码修复的问题（例如，第三方 widget bug）为 "延迟"，不管层级。

## 阶段 8: 修复循环

对于每个可修复的问题，按严重性顺序：

### 8a. 定位源

```bash
# grep 错误消息、组件名、路由定义
# 为受影响页面匹配的文件 glob
```

- 查找负责 bug 的源文件
- 只修改与问题直接相关的文件

### 8b. 修复

- 读源代码，理解上下文
- 做**最小修复** — 解决问题的最小更改
- 不重构周围代码，不添加特性，不 "改进" 无关的东西

### 8c. 提交

```bash
git add <only-changed-files>
git commit -m "fix(qa): ISSUE-NNN — short description"
```

- 每个修复一个提交。永远不要束多个修复。
- 消息格式：`fix(qa): ISSUE-NNN — short description`

### 8d. 重新测试

导航回受影响的页面并验证修复：

```bash
$B goto <affected-url>
$B screenshot "$REPORT_DIR/screenshots/issue-NNN-after.png"
$B console --errors
$B snapshot -D
```

采取**修复前/修复后屏幕截图对** 对每个修复。

### 8e. 分类

- **verified**: 重新测试确认修复有效，没有新错误引入
- **best-effort**: 修复应用但无法完全验证
- **reverted**: 检测到回归 → `git revert HEAD` → 标记问题为 "延迟"

### 8e.5. 回归测试

跳过如果：分类不是 "verified"，或修复纯粹是 CSS 没有 JS 行为，或没有检测到测试框架并且用户拒绝了引导。

**研究现有的测试模式：**

读 2-3 个最接近修复的测试文件（同一目录，同一代码类型）。完全匹配：
- 文件命名、导入、断言样式、describe/it 嵌套、设置/拆卸模式

**追踪 bug 的代码路径，然后写一个回归测试：**

在写测试前，追踪通过修复的代码的数据流：
- 什么输入/状态触发了 bug？（确切的前置条件）
- 它跟踪什么代码路径？
- 它在哪里破裂？
- 什么其他输入可以击中同样的代码路径？（边界情况）

测试**必须**：
- 设置触发 bug 的前置条件
- 执行暴露 bug 的操作
- 声言正确行为（不是 "它呈现" 或 "它不抛出"）
- 如果找到周围边界情况时，测试那些也
- 包括完整的访问注释

**测试类型决定：**
- 控制台错误 / JS 异常 / 逻辑 bug → 单元或集成测试
- 破碎的表单 / API 失败 / 数据流 bug → 集成测试
- 破碎的 dropdown / 动画 / 交互状态 → 组件测试
- 纯 CSS → 跳过

**使用自动递增名字避免碰撞：** 检查现有 `{name}.regression-*.test.{ext}` 文件，采取 max 号 + 1。

**运行仅新测试文件：**

```bash
{detected test command} {new-test-file}
```

**评估：**
- 通过 → 提交：`git commit -m "test(qa): regression test for ISSUE-NNN — {desc}"`
- 失败 → 修复一次。仍然失败 → 删除测试，延迟。
- 探索 > 2 分钟 → 跳过并延迟。

### 8f. 自我调节

每 5 个修复（或任何还原后），计算 WTF 可能性：

```
WTF-LIKELIHOOD:
  从 0% 开始
  每个还原：                +15%
  每个接触 > 3 个文件的修复: +5%
  修复 15 后：               +1% 每额外修复
  所有剩余低严重性：       +10%
  接触无关文件：           +20%
```

**如果 WTF > 20%：** 立即停止。向用户显示迄今为止所做的。问是否继续。

**硬限制：30 个修复。** 30 个修复后停止，不管剩余问题。

## 阶段 10: 报告

写报告到本地和项目范围位置：

**本地：** `.gstack/qa-reports/qa-report-{domain}-{YYYY-MM-DD}.md`

**项目范围：**
```bash
eval "$(~/.claude/skills/gstack/bin/gstack-slug 2>/dev/null)" && mkdir -p ~/.gstack/projects/$SLUG
```
写到 `~/.gstack/projects/{slug}/{user}-{branch}-test-outcome-{datetime}.md`

**每个问题添加** （超过标准报告模板）：
- 修复状态：verified / best-effort / reverted / deferred
- 提交 SHA（如果修复）
- 文件更改（如果修复）
- 修复前/修复后屏幕截图（如果修复）

**摘要部分：**
- 发现的总问题
- 应用的修复（verified: X, best-effort: Y, reverted: Z）
- 延迟的问题
- 健康分数增量：基准 → 最后

**PR 摘要：** 包括适合 PR 描述的一行摘要：
> "QA 发现 N 个问题，修复了 M 个，健康分数 X → Y。"

## 额外规则（qa 专用）

11. **干净的工作树必需。** 如果很脏，在继续前使用 AskUserQuestion 提供提交/隐藏/中止。
12. **每个修复一个提交。** 永远不要束多个修复进一个提交。

15. **自我调节。** 跟踪 WTF 可能性启发式。在疑问时停止并问。
