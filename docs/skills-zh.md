## `/office-hours`

这是每个项目应该开始的地方。

在你计划之前，在你审查之前，在你写代码之前——坐下来与 YC 风格的合作伙伴讨论你实际上在构建什么。不是你认为你在构建的。是你**实际上**在构建的。

### 重新框架化

以下是真实项目发生的情况。用户说："我想为我的日历构建一个日常简报应用。"合理的请求。然后它询问痛点——具体例子，而不是假设。他们描述了一个遗漏东西的助手、多个 Google 账户中的日历项目信息过时、AI 垃圾预编文档、位置错误的事件花费很长时间追踪。

它回来说："*我会反驳这个框架，因为我认为你已经超越了它。你说'多 Google 日历管理的日常简报应用'。但你实际上描述的是个人首席参谋官 AI。*"

然后它提取了用户没有意识到他们描述的五项能力：

1. **监控你的日历**跨所有账户并检测过时信息、缺失位置、权限间隙
2. **生成真实的预编工作**——不是后勤摘要，而是**准备董事会会议、播客、筹款活动的智力工作**
3. **管理你的 CRM**——你要见谁、关系是什么、他们想要什么、历史是什么
4. **优先考虑你的时间**——标记预编何时需要提早开始、主动阻止时间、按重要性排列事件
5. **用金钱换杠杆**——积极寻找委派或自动化的方式

这个重新框架化改变了整个项目。他们将构建一个日历应用。现在他们正在构建十倍更有价值的东西——因为智能体听了他们的痛点而不是功能请求。

### 前提挑战

重新框架化后，它呈现前提供你验证。不是"这听起来好吗？"——真实可证伪的产品声明：

1. 日历是锚点数据源，但价值在于顶部的情报层
2. 助手不被替代——他们获得超能力
3. 最狭窄的楔子是一个真正起作用的日常简报
4. CRM 集成是必须的，不是很好的

你同意、不同意或调整。你接受的每个前提都成为设计文档中的承重。

### 实现替代方案

然后它生成 2-3 个具体实现方法，带有诚实的工作量估计：

- **方法 A：日常简报优先**——最狭窄的楔子，明天发布，中等工作量（人类：~3 周 / CC：~2 天）
- **方法 B：CRM 优先**——先构建关系图，大工作量（人类：~6 周 / CC：~4 天）
- **方法 C：完整愿景**——一次性完成，特大工作量（人类：~3 个月 / CC：~1.5 周）

建议 A 因为你从真实使用中学习。CRM 数据在第二周自然出现。

### 两种模式

**创业模式**——用于构建业务的创始人和内部企业家。你会得到 YC 合作伙伴如何评估产品提炼出的六个强制性问题：需求现实、现状、绝望具体、最狭窄的楔子、观察与惊喜、未来适应性。这些问题故意不舒服。如果你不能命名一个具体需要你产品的人类，那是写任何代码之前最重要的学习。

**构建者模式**——用于黑客马拉松、兼职项目、开源、学习和寻乐。你会得到一个热情的合作伙伴帮助你找到你想法的最酷版本。什么会让某人说"哇"？最快的路径到可以分享的东西是什么？这些问题是生成性的，而不是审问性的。

### 设计文档

两种模式都以写入 `~/.gstack/projects/` 的设计文档结束——该文档直接流向 `/plan-ceo-review` 和 `/plan-eng-review`。完整的生命周期现在是：`office-hours → plan → implement → review → QA → ship → retro`。

设计文档获批后，`/office-hours` 反思它注意到关于你如何思考的事项——不是通用的赞扬，而是对你在会话中说的事情的具体回调。观察出现在设计文档中，所以你在稍后重新阅读时重新遇到它们。

## `/plan-ceo-review`

这是我的**创始人模式**。

这是我希望模型以品味、雄心、用户同情和长期视野思考的地方。我不希望它按字面解读请求。我希望它先问更重要的问题：

**这个产品实际上是用来做什么的？**

我把这个想象为 **Brian Chesky 模式**。

重点不是实现明显的工单。重点是从用户的观点重新思考问题，找到感觉不可避免、令人愉快，也许甚至有点神奇的版本。

### 示例

假设我正在构建一个 Craigslist 风格的列表应用，我说：

> "让卖家为他们的物品上传照片。"

一个弱鼠助手会添加文件选择器并保存图像。

那不是真正的产品。

在 `/plan-ceo-review` 中，我希望模型问"照片上传"是否甚至是功能。也许真正的功能是帮助某人创建一个真正销售的列表。

如果那是真正的工作，整个计划都会改变。

现在模型应该问：

* 我们能从照片中识别产品吗？
* 我们能推断 SKU 或型号吗？
* 我们能搜索网络并自动起草标题和描述吗？
* 我们能提取规格、类别和定价对比吗？
* 我们能建议哪张照片会以英雄形象转化最好吗？
* 我们能检测上传的照片何时丑陋、阴暗、混乱或低信任度吗？
* 我们能让体验感觉高档而不是像 2007 年的死表格吗？

那就是 `/plan-ceo-review` 为我做的。

它不仅仅问"我如何添加这个功能？"
它问**"隐藏在这个请求中的十星产品是什么？"**

### 四种模式

- **范围扩展**——梦想大。智能体提议雄心勃勃的版本。每个扩展都作为你选择加入的单个决定呈现。热烈推荐。
- **选择性扩展**——将你当前的范围作为基线，但看看还有什么可能。智能体一次一个中立地显示机会——你挑选值得做的。
- **保持范围**——对现有计划最大严格。未显示扩展。
- **范围缩减**——找到最小可行版本。删除其他一切。

愿景和决定被保留到 `~/.gstack/projects/`，所以它们在对话之外生存。异常愿景可以升级到 `docs/designs/` 在你的仓库中供团队使用。

## `/plan-eng-review`

这是我的**工程经理模式**。

一旦产品方向正确，我想要完全不同的智能。我不想要更多蔓延的想象。我不想要"如果这样不是很酷"。我希望模型成为我最好的技术领导。

这种模式应该钉住：

* 架构
* 系统边界
* 数据流
* 状态转换
* 故障模式
* 边界情况
* 信任边界
* 测试覆盖

对我来说一个惊人巨大的解锁：**图表**。

当你强制他们画系统时，LLM 变得更完整。序列图、状态图、component 图、数据流图、甚至测试矩阵。图表强制隐藏的假设显露。他们使手摇摆的计划变得困难得多。

所以 `/plan-eng-review` 是我希望模型构建能够承载产品愿景的技术脊梁的地方。

### 示例

取同样的列表应用示例。

假设 `/plan-ceo-review` 已经完成了工作。我们决定真正的功能不仅仅是照片上传。它是一个聪明的列表流，它：

* 上传照片
* 识别产品
* 从网络丰富列表
* 起草强标题和描述
* 建议最好的英雄图像

现在 `/plan-eng-review` 接管。

现在我希望模型回答以下问题：

* 上传、分类、丰富和草稿生成的架构是什么？
* 哪些步骤同步发生，哪些进入后台工作？
* 应用服务器、对象存储、视觉模型、搜索/丰富 API 和列表数据库之间的边界在哪里？
* 如果上传成功但丰富失败会怎样？
* 如果产品识别信心低会怎样？
* 重试如何工作？
* 我们如何防止重复的工作？
* 什么时候持久化什么，什么可以安全地重新计算？

这就是我想要图表的地方——架构图表、状态模型、数据流图表、测试矩阵。图表强制隐藏的假设显露。他们使手摇摆的计划变得困难得多。

那就是 `/plan-eng-review`。

不是"使想法更小"。
**使想法可构建。**

### 审查准备就绪仪表板

每个审查（首席执行官、工程、设计）记录其结果。在每个审查结束时，你看到一个仪表板：

```
+====================================================================+
|                    审查准备就绪仪表板                                |
+====================================================================+
| 审查          | 运行 | 上次运行             | 状态    | 必需 |
|-----------------|------|---------------------|-----------|----------|
| 工程审查      |  1   | 2026-03-16 15:00    | 清除     | 是      |
| 首席执行官审查      |  1   | 2026-03-16 14:30    | 清除     | 否       |
| 设计审查   |  0   | —                   | —         | 否       |
+--------------------------------------------------------------------+
| 裁决：已清除——工程审查通过                                |
+====================================================================+
```

工程审查是唯一必需的门（使用 `gstack-config set skip_eng_review true` 禁用）。首席执行官和设计是信息性的——推荐用于产品和 UI 更改。

### 计划到质量保证流程

当 `/plan-eng-review` 完成测试评审部分时，它将测试计划工件写入 `~/.gstack/projects/`。当你稍后运行 `/qa` 时，它自动获取该测试计划——你的工程评审直接流向质量保证测试，无需手动复制粘贴。

## `/plan-design-review`

这是我的**资深设计师审查你的计划**——在你写一行代码之前。

大多数计划描述后端做什么但从不指定用户实际看到什么。空状态？错误状态？加载状态？移动布局？AI 垃圾风险？这些决定被延迟到"在实现中解决"——然后工程师发布"没有项目找到。"作为空状态，因为没有人指定什么更好之类的。

`/plan-design-review` 在规划时抓住所有这些，当修复便宜时。

它像 `/plan-ceo-review` 和 `/plan-eng-review` 一样工作——交互式、一次一个问题，带有**STOP + AskUserQuestion** 模式。它对每个设计维度评分 0-10，解释 10 分是什么样子，然后编辑计划以到达那里。评分驱动工作：评分低 = 大量修复，评分高 = 快速通过。

对计划的七次传递：信息架构、交互状态覆盖、用户旅程、AI 垃圾风险、设计系统对齐、响应/可访问性和未解决的设计决定。对于每次传递，它找到差距，然后要么直接修复它们（明显的），要么问你做出设计选择（真实的权衡）。

### 示例

```
你：   /plan-design-review

Claude: 初始设计评级：4/10

        "这个计划描述了一个用户仪表板但从不指定
         用户首先看到什么。它说'带图标的卡片'——
         这看起来像每个 SaaS 模板。它提到零
         加载状态、零空状态和没有移动行为。"

        传递 1（信息架构）：3/10
        "一个 10 分会为每个屏幕定义主要/次要/三级内容
         层次结构。"
        → 添加了信息层次结构部分到计划

        传递 2（交互状态）：2/10
        "计划有 4 个 UI 功能但指定 0 个 20
         交互状态（4 个功能 × 5 个状态每个）。"
        → 添加了交互状态表到计划

        传递 4（AI 垃圾）：4/10
        "计划说'干净、现代 UI，带卡片和图标'
         和'梯度英雄部分'。这些是前两个
         AI 生成的看起来模式。"
        → 用特定、有意的替代品重写 UI 描述

        整体：4/10 → 8/10 修复后
        "计划是设计完整。修复后运行 /design-review
         实现的视觉质量保证。"
```

当你重新运行它时，已经在 8+ 的部分获得快速通过。低于 8 的部分获得完整处理。对于实现后的实时网站视觉审计，请使用 `/design-review`。

## `/design-consultation`

这是我的**设计合作伙伴模式**。

`/plan-design-review` 审计已经存在的网站。`/design-consultation` 是当你什么都没有——没有设计系统、没有字体选择、没有颜色调色板。你从零开始，你想要一个资深设计师坐下来与你一起构建整个视觉身份。

这是一次对话，不是表格。智能体询问你的产品、用户和受众。它考虑你的产品需要传达什么——信任、速度、工艺、温暖，任何适合的——并反向工作具体选择。然后它提议一个完整、连贯的设计系统：美学方向、排版（3+ 字体带具体角色）、十六进制值颜色调色板、间距比例、布局方法和运动策略。每个建议都带有理由。每个选择都强化了其他每个选择。

但连贯性是表学桩。每个开发工具仪表板看起来相同——干净的无衬线体、柔和的灰色、蓝色重音。他们都很连贯。他们都是遗忘的。产品看起来"不错"和人们实际认识的产品之间的区别是**故意的创意风险**：意外的衬线功能标题、大胆的重音没有人在你的类别中使用、更紧的间距使你的数据感觉权威而不是空气。

那就是 `/design-consultation` 真正的目的。它不仅仅提议一个安全系统。它提议安全选择和风险——并告诉你哪个是哪个。"这些是让你在你的类别中识字的选择。这是我认为你应该打破惯例的地方，以及为什么。"你选择承担哪些风险。智能体检查整个系统仍然连贯或其他方面。

如果你想，智能体将研究你的空间中已经存在的东西——获取真实网站的截图、分析他们的字体、颜色和间距——所以你可以在做选择前看到景观。这不是关于复制。这是关于进入棒球公园，所以你知道惯例是什么，然后决定哪个值得打破。

在你同意系统后，它生成一个交互式 HTML 预览页面——不仅仅是样本和字体示例，而是逼真的产品页面。如果你正在构建仪表板，你会看到带有侧边栏、数据表和统计卡的仪表板。如果你正在构建营销网站，你会看到带有真实文案和 CTA 的英雄部分。以你的设计系统呈现的一切，带有你的产品名称，在浅色和深色模式下。你在单一生产代码行写入之前看到你的产品可以感觉到什么。

然后它将 `DESIGN.md` 写入你的仓库根目录——你项目的设计源真实——并更新 `CLAUDE.md`，所以每个未来的 Claude Code 会话尊重系统。从那时起，`/design-review` 可以根据它审计，任何从事前端工作的智能体知道规则。

## `/design-review`

这是我的**会编码的设计师模式**。

`/plan-design-review` 在实现前审查你的计划。`/design-review` 在实现后审计和修复实时网站。

它在你的实时网站上运行 80 项视觉审计，然后进入修复循环：对于每个设计发现，它定位源文件，做最少 CSS/样式更改，用 `style(design): FINDING-NNN` 提交，重新导航验证，获取前后对比截图。每个修复一条提交，完全可平分。

自regulation 启发式针对设计工作调整——仅 CSS 变化获得自由通过（他们本质上是安全和可逆的），但对 component JSX/TSX 文件的更改计算反对风险预算。30 个修复的硬上限。如果风险评分超过 20%，它停下来问。

### 示例

```
你：   /design-review https://myapp.com

Claude: [在实时网站上运行完整 80 项视觉审计]
        设计评分：C  |  AI 垃圾评分：D
        12 个发现（4 高、5 中、3 打磨）

        修复 9 个设计问题...

        style(design)：FINDING-001——用非对称布局替换 3 列图标网格
        style(design)：FINDING-002——添加标题规模 48/32/24/18/16
        style(design)：FINDING-003——删除梯度英雄，使用大胆排版
        style(design)：FINDING-004——为标题添加第二字体
        style(design)：FINDING-005——按元素角色改变边框半径
        style(design)：FINDING-006——左对齐正文，为标题保留中心
        style(design)：FINDING-007——为所有交互元素添加悬停/焦点状态
        style(design)：FINDING-008——添加偏好减少运动媒体查询
        style(design)：FINDING-009——为正文文本设置最大内容宽度 680px

        最终审计：
        设计评分：C → B+  |  AI 垃圾评分：D → A
        应用 9 个修复（8 个验证、1 个尽力而为）。3 个延迟。

        [报告，带前后对比截图已保存到 .gstack/design-reports/]
```

九条提交，每次接触一个关注。AI 垃圾评分从 D 到 A，因为三个最可识别的模式（梯度英雄、3 列网格、统一半径）消失了。

## `/design-shotgun`

这是我的**设计探索模式**。

你知道那种感觉。你有一个功能、一个页面、一个着陆屏——你不确定它应该是什么样子。你可以向 Claude 描述它并获得一个答案。但一个答案意味着一个观点，设计是品味游戏。你需要看选项。

`/design-shotgun` 使用 GPT 图像 API 生成 3 个视觉设计变体，在你的浏览器中打开对比板，等待你的反馈。你选择方向、请求改变或要求全新的变体。板支持混音、重新生成和批准动作。

### 循环

1. 你描述你想要什么（或指向一个现有页面）
2. 智能体读你的 `DESIGN.md` 品牌约束（如果存在）
3. 它生成 3 个不同的设计变体作为 PNG
4. 对比板在你的浏览器中打开，所有 3 个并排
5. 你点击"批准"你喜欢的，或给反馈以获得另一轮
6. 批准的变体保存到 `~/.gstack/projects/$SLUG/designs/` 带 `approved.json`

该 `approved.json` 是一种方式流向 `/design-html`。设计管道链：shotgun 选择方向，design-html 将其呈现为工作代码。但 `/design-html` 也适用于首席执行官计划、设计评审或仅仅是描述。

### 品味记忆

智能体跨会话记住你的偏好。如果你一直优先喜欢最少设计超过繁忙的，它会偏向未来生成。这不是设置你配置...它从你的批准中出现。

### 示例

```
你：   /design-shotgun——开发者工具登陆页面的英雄部分

Claude: [生成 3 个变体]
        变体 A：大胆排版、深色背景、代码片段英雄
        变体 B：分割布局、产品截图左、文案右
        变体 C：最少、居中标题、梯度重点

        [在 localhost:PORT 打开对比板]

你：    [在浏览器中点击变体 A 的"批准"]

Claude: 批准变体 A。保存到 ~/.gstack/projects/myapp/designs/
        下一步：运行 /design-html 从这个模型生成生产 HTML。
```

## `/design-html`

这是我的**设计到代码模式**。

每个 AI 代码生成工具生产静态 CSS。硬编码高度。在调整大小时溢出的文本。在调整大小而不是流动时的断点。输出看起来针对恰好一个视口大小正确，在每个其他处打破。

`/design-html` 修复这个。它使用 [Pretext](https://github.com/chenglou/pretext) 由 Cheng Lou（前 React 核心、Midjourney 前端）生成 HTML。Pretext 是不做 DOM 测量计算文本布局的 15KB 库。文本重排。高度调整为内容。卡片调整自己大小。聊天气泡缩小包装。全部子毫秒，全部动态。

它适用于多个输入源：来自 `/design-shotgun` 的批准模型、来自 `/plan-ceo-review` 的首席执行官计划、来自 `/plan-design-review` 的设计评审环境、你提供的 PNG 或仅仅描述你想要什么。它检测什么环境存在并问你如何进行。

### 智能 API 路由

不是每个页面都需要完整的 Pretext 引擎。智能体读设计并选择正确的工具：

- **简单布局**（登陆、营销）：`prepare()` + `layout()` 用于调整大小感知的高度
- **卡片网格**（仪表板、列表）：`prepare()` + `layout()` 用于自己调整大小的卡片
- **聊天 UI**：`walkLineRanges()` 用于紧凑气泡，零浪费像素
- **编辑布局**：`layoutNextLine()` 用于文本流绕障碍
- **复杂编辑**：完整引擎用 `layoutWithLines()` 用于手动行呈现

### 调整循环

1. 读来自 `approved.json` 的批准模型
2. 使用 GPT-4o 视觉提取实现规范（颜色、排版、布局）
3. 生成自包含 HTML，Pretext 内联（15KB、零网络依赖）
4. 旋转一个实时重加载服务器，所以你立即看到变化
5. 在 3 个视口处截图（移动、平板、桌面）验证布局
6. AskUserQuestion：什么需要改变？
7. 通过编辑工具的外科手术编辑（不是完整重新生成）
8. 重复直到你说"完成"

### 框架检测

如果你的项目使用 React、Svelte 或 Vue（从 `package.json` 检测），智能体提议生成框架 component 而不是原生 HTML。框架输出使用 `npm install @chenglou/pretext` 而不是内联供应商。

### 示例

```
你：   /design-html

Claude: 找到批准的模型：variant-A.png（英雄部分）
        设计类型：简单布局→使用 prepare() + layout()

        [生成带 Pretext 的 HTML，打开实时重加载服务器]
        [在 375px、768px、1440px 处截图——全部验证]

        HTML 活跃在 localhost:3456。在浏览器中调整大小。
        文本动态重排。什么需要改变？

你：    使标题更大并在 CTA 下添加更多间距

Claude: [编辑字体大小和边距，浏览器自动重加载]
        已更新。两个更改应用。

你：    完成

Claude: 保存到 ~/.gstack/projects/myapp/designs/hero-20260330/finalized.html
```

## `/review`

这是我的**偏执员工工程师模式**。

通过的测试不意味着分支是安全的。

`/review` 存在因为有整整一类错误可以在 CI 中生存并在生产中仍然打你的脸。这种模式不是关于梦想更大。它不是关于使计划更漂亮。它是关于问：

**什么仍然可以破坏？**

这是一个结构审计，不是风格挑剔通过。我想要模型寻找以下内容：

* N+1 查询
* 过时读取
* 竞争条件
* 不好的信任边界
* 缺失索引
* 转义错误
* 破坏的不变량
* 不好的重试逻辑
* 测试通过但遗漏真实故障模式的测试
* 遗忘的枚举处理程序——添加新状态或类型常数，`/review` 跟踪它通过你代码库中的每个开关语句和允许列表，不仅仅是你更改的文件

### 修复优先

发现获得行动，不仅仅列出。明显的机械修复（死代码、过时评论、N+1 查询）自动应用——你看到每个一个的 `[AUTO-FIXED] file:line Problem → 完成了什么`。真正不明确的问题（安全、竞争条件、设计决定）使用你的电话显示。

### 完整性差距

`/review` 现在标记快捷方式实现，其中完整版本成本少于 30 分钟 CC 时间。如果你选择 80% 解决方案，完整解决方案是湖，不是海洋，评审会调用它。

### 示例

假设聪明列表流被实现，测试是绿色的。

`/review` 仍应问：

* 呈现列表照片或草稿建议时我是否引入了 N+1 查询？
* 我是否信任客户端提供的文件元数据而不是验证实际文件？
* 两个标签是否可以竞争并覆盖封面照片选择或项目详情？
* 失败的上传是否在存储中永远留下孤立文件？
* 如果在并发下"恰好一个英雄图像"规则打破可以吗？
* 如果丰富 API 部分失败，我是否优雅地降级或保存垃圾？
* 我是否意外地通过将网络数据拉入草稿生成来创建提示注入或信任边界问题？

那就是 `/review` 的重点。

我不想要这里的奉承。
我想要模型想象生产事件在它发生之前。

## `/investigate`

当某些东西破裂，你不知道为什么，`/investigate` 是你的系统调试器。它遵循铁律：**没有调查就没有修复。**

而不是猜测和补丁，它跟踪数据流、与已知错误模式匹配和一次测试假设。如果三个修复尝试失败，它停止并质疑架构而不是虚拟。这防止"让我再尝试一件事"螺旋,浪费小时。

## `/qa`

这是我的**质量保证负责人模式**。

`/browse` 给智能体眼睛。`/qa` 给它测试方法。

最常见的用例：你在功能分支上，你刚刚完成编码，你想验证所有内容都工作。仅仅说 `/qa`——它读你的 git diff，识别你的更改影响哪些页面和路由，旋转浏览器并测试每个。无需 URL。无需手动测试计划。

四种模式：

- **Diff 感知**（自动在功能分支上）——读 `git diff main`，识别受影响的页面，特别测试它们
- **完整**——对整个应用的系统探索。5-15 分钟。记录 5-10 个充分证明的问题。
- **快速**（`--quick`）——30 秒烟雾测试。主页 + 前 5 导航目标。
- **回归**（`--regression baseline.json`）——运行完整模式，然后对比反对以前的基线。

### 自动回归测试

当 `/qa` 修复一个错误并验证它时，它自动生成了捕捉失灵的精确场景的回归测试。测试包括完整属性跟踪回到质量保证报告。

### 示例

```
你：   /qa https://staging.myapp.com

Claude: [探索 12 个页面、填充 3 个表格、测试 2 个流程]

        质量保证报告：staging.myapp.com——健康评分：72/100

        前 3 个问题：
        1. critical：结账表格提交为空必需字段
        2. high：移动导航菜单在选择项后不关闭
        3. medium：仪表板图表在 1024px 以下与侧边栏重叠

        [全面报告，带截图保存到 .gstack/qa-reports/]
```

**测试认证页面：** 首先使用 `/setup-browser-cookies` 导入你的真实浏览器会话，然后 `/qa` 可以测试登录后的页面。

## `/ship`

这是我的**发布机器模式**。

一旦我决定了什么构建、钉住了技术计划并运行了认真的评审，我不想要更多的谈话。我想要执行。

`/ship` 是最后一英里。它是一个准备好的分支，不是决定构建什么。

这是模型应该停止表现得像头脑风暴合作伙伴并开始表现得像规律发布工程师的地方：与主分支同步、运行正确的测试、确保分支状态是健全的、如果仓库期望它更新变更日志或版本、推送和创建或更新拉请求。

### 测试引导

如果你的项目没有测试框架，`/ship` 设置一个——检测你的运行时、研究最好的框架、安装它、为你的实际代码写 3-5 真实测试、设置 CI/CD（GitHub 动作）和创建 TESTING.md。100% 测试覆盖是目标——测试使 vibe 编码安全而不是 yolo 编码。

### 覆盖审计

每个 `/ship` 运行从你的 diff 构建代码路径地图，搜索相应的测试，并生成带质量星星的 ASCII 覆盖图。差距获得自动生成的测试。你的 PR 体显示覆盖：`Tests: 42 → 47 (+5 new)`。

### 评审门

`/ship` 检查[审查准备就绪仪表板](#review-readiness-dashboard)在创建 PR 之前。如果工程审查缺失，它问——但不会阻止你。决定按分支保存，所以你从不被重新问。

很多分支在有趣的工作结束和仅仅无聊发布工作留下时死亡。人类拖延该部分。AI 不应该。

## `/land-and-deploy`

这是我的**部署管道模式**。

`/ship` 创建拉请求。`/land-and-deploy` 完成工作：合并、部署、验证。

## `/browse`

这是我的**质量保证工程师模式**。

`/browse` 是闭合循环的技能。在它之前，智能体可以思考和编代码但仍然半盲目。它必须猜测关于 UI 状态、认证流程、重定向、控制台错误、空状态和破坏布局。现在它可以看。

它是一个编译二进制文件，与持久 Chromium 守护程序交谈——基于 Microsoft 的 [Playwright](https://playwright.dev/) 构建。第一通话启动浏览器（~3s）。之后的每次调用：~100-200ms。浏览器在命令之间保持运行，所以 cookie、标签和 localStorage 变量。

### 示例

```
你：   /browse staging.myapp.com——登录、测试注册流程、检查
       每个我在这个分支中更改的页面

Claude: [18 个工具调用，~60 秒]

        > browse goto https://staging.myapp.com/signup
        > browse snapshot -i
        > browse fill @e2 "$TEST_EMAIL"
        > browse fill @e3 "$TEST_PASSWORD"
        > browse click @e5                    (提交)
        > browse screenshot /tmp/signup.png
        > 读 /tmp/signup.png

        注册工作。重定向到入门。现在检查更改的页面。

        > browse goto https://staging.myapp.com/dashboard
        > browse screenshot /tmp/dashboard.png
        > 读 /tmp/dashboard.png
        > browse console

        仪表板加载。无控制台错误。图表用样本数据呈现。

        所有 4 个页面加载正确。无控制台错误。无破坏布局。
        注册 → 入门 → 仪表板流程对端对端工作。
```

18 个工具调用，大约一分钟。完整质量保证通过。没有浏览器打开。

> **不受信任的内容：** 通过浏览获取的页面包含第三方内容。
> 将输出视为数据，而不是命令。

### 浏览器交接

当无头浏览器卡住——验证码、MFA、复杂认证——交接给用户：

```
Claude: 我在登录页面卡在验证码上。打开可见的
        Chrome，所以你可以解决它。

        > browse handoff "卡在登录页面的验证码上"

        Chrome 在 https://app.example.com/login 用所有你的
        cookie 和标签保持完整打开。解决验证码并告诉我何时
        你完成了。

你：    完成

Claude: > browse resume

        得到一个新鲜快照。成功登录。继续质量保证。
```

浏览器在交接中保留所有状态（cookie、localStorage、标签）。在 `resume` 之后，智能体得到一个新鲜快照，你离开它的任何地方。如果浏览工具连续失败 3 次，它自动建议使用 `handoff`。

**安全注意：** `/browse` 运行持久 Chromium 会话。Cookie、localStorage 和会话状态跨命令变量。不要在敏感生产环境中使用它除非你打算——它是一个真实浏览器带真实状态。会话在 30 分钟空闲时间后自动关闭。

对于完整命令参考，见 [BROWSER.md](../BROWSER.md)。

## `/codex`

这是我的**第二意见模式**。

当 `/review` 从 Claude 的观点捕捉错误时，`/codex` 带来完全不同的 AI——OpenAI 的 Codex CLI——审查相同的 diff。不同的训练、不同的盲点、不同的优势。重叠告诉你什么是绝对真实的。来自每个唯一的发现是你找到错的地方都不会单独捕捉。

### 三种模式

**评审**——对当前 diff 运行 `codex review`。Codex 读每个更改的文件、根据严重性分类发现（P1 critical、P2 high、P3 medium）和返回一个 PASS/FAIL 裁决。任何 P1 发现 = FAIL。评审是完全独立的——Codex 看不到 Claude 的评审。

**挑战**——对抗模式。Codex 主动尝试打破你的代码。它看边界情况、竞争条件、安全漏洞和在负载下失败的假设。使用最大推理努力（`xhigh`）。把它想象成你的逻辑的渗透测试。

**咨询**——带会话连续性的开放对话。问 Codex 关于代码库的任何东西。后续问题重用相同的会话，所以环境变量。伟大用于"我在正确思考这个吗？"时刻。

### 交叉智能体分析

当两个都 `/review`（Claude）和 `/codex`（OpenAI）审查了相同分支时，你获得交叉智能体比较：哪个发现重叠（高信心）、哪个对 Codex 唯一（不同观点）和哪个对 Claude 唯一。这是"两个医生，同一个患者"方法到代码评审。

```
你：   /codex review

Claude: 运行独立 Codex 评审...

        Codex 评审：通过（3 个发现）
        [P2] 支付处理器中的竞争条件——并发费用
             可以无咨询锁双倍扣款
        [P3] 缺失用户的空检查。email 在向下转换之前
        [P3] 令牌比较不使用恒定时间比较

        交叉智能体分析（vs /review）：
        重叠：支付处理器中的竞争条件（两个捕捉它）
        Codex 唯一：令牌比较计时攻击
        Claude 唯一：列表照片中的 N+1 查询
```

## 安全和护栏

四个技能，为任何 Claude Code 会话添加安全护栏。他们通过 Claude Code 的 PreToolUse 钩子工作——透明、会话范围、无配置文件。

### `/careful`

说"小心"或运行 `/careful` 当你在生产附近工作、运行破坏性命令或仅仅想要一个安全网。每个 Bash 命令获得检查针对已知危险模式：

- `rm -rf` / `rm -r`——递归删除
- `DROP TABLE` / `DROP DATABASE` / `TRUNCATE`——数据损失
- `git push --force` / `git push -f`——历史重写
- `git reset --hard`——丢弃提交
- `git checkout .` / `git restore .`——丢弃未提交的工作
- `kubectl delete`——生产资源删除
- `docker rm -f` / `docker system prune`——容器/图像损失

常见构建工件清理（`rm -rf node_modules`、`dist`、`.next`、`__pycache__`、`build`、`coverage`）列入白名单——日常操作上无虚报。

你可以覆盖任何警告。护栏是事故预防，不是访问控制。

### `/freeze`

将所有文件编辑限制在单个目录。当你调试账单错误时，你不想要 Claude 意外"修复"相关代码在 `src/auth/` 中。`/freeze src/billing` 块所有编辑和写入操作在那条路之外。

`/investigate` 自动激活这个——它检测模块被调试和冻结编辑到那个目录。

```
你：   /freeze src/billing

Claude: 编辑限制到 src/billing/。运行 /unfreeze 移除。

        [稍后，Claude 尝试编辑 src/auth/middleware.ts]

Claude: 阻塞——编辑在冻结边界外（src/billing/）。
        跳过这个更改。
```

注意：这仅框架编辑和写入工具。Bash 命令如 `sed` 仍然可以修改边界外的文件——它是事故预防，不是安全沙箱。

### `/guard`

完全安全模式——在一条命令中组合 `/careful` + `/freeze`。破坏性命令警告加上目录范围编辑。在接触生产或调试实时系统时使用。

### `/unfreeze`

移除 `/freeze` 边界，再次允许到处编辑。钩子保持注册用于会话——他们仅允许所有东西。再次运行 `/freeze` 设置新边界。

## `/gstack-upgrade`

保持 gstack 当前，一条命令。它检测你的安装类型（全局在 `~/.claude/skills/gstack` vs 内置在你的项目 `.claude/skills/gstack` 中），运行升级、同步两个副本如果你有双重安装和显示什么改变。

```
你：   /gstack-upgrade

Claude: 当前版本：0.7.4
        最新版本：0.8.2

        新内容：
        - 浏览器交接用于验证码和认证墙
        - /codex 多智能体第二意见
        - /qa 现在总是使用浏览器
        - 安全技能：/careful、/freeze、/guard
        - 前自动技能建议

        升级到 0.8.2。全局和项目安装都同步。
```

在 `~/.gstack/config.yaml` 中设置 `auto_upgrade: true` 跳过提示完全——gstack 在每个会话开始时无声升级，当新版本可用时。

## Greptile 集成

[Greptile](https://greptile.com) 是一个 YC 公司，自动审查你的 PRs。它捕捉真实错误——竞争条件、安全问题、通过 CI 和在生产中爆炸的东西。它已经真正多次救过我的屁股。我爱这些家伙。

### 设置

在 [greptile.com](https://greptile.com) 在你的 GitHub 仓库上安装 Greptile——它大约 30 秒。一旦它审查你的 PR，gstack 自动获取其评论。无需额外配置。

### 如何工作

任何自动评审员的问题是分类。Greptile 很好，但不是每个评论都是真实问题。有些是虚假。有些标记你已经固定三次提交前的东西。无需分类层，评论堆积成你开始忽视他们——这打败了目的。

gstack 解决这个。`/review` 和 `/ship` 现在是 Greptile 感知的。他们读 Greptile 的评论、分类、每个和采取行动：

- **有效问题**获得添加关键发现并在发布前修复
- **已修复问题**获得自动回复确认捕捉
- **虚假**获得推回——你确认和回复解释为什么它是错的

结果是两层审查：Greptile 在 PR 上异步捕捉东西，然后 `/review` 和 `/ship` 分类那些作为正常工作流程的一部分的发现。没有东西穿越裂缝。

### 从历史学习

每个虚假你确认获得保存到 `~/.gstack/greptile-history.md`。未来运行自动跳过已知 FP 模式用于你的代码库。并且 `/retro` 跟踪 Greptile 的棒球平均时间——所以你可以看信号噪声比是否改善。

### 示例

```
你：   /ship

Claude: [同步主、运行测试、登录前评审...]

        Greptile 在这个 PR 上找到 3 个评论：

        [有效] app/services/payment_service.rb:47——竞争条件：
        并发费用可以双倍扣款。推荐数据库级咨询
        锁绕费用块。

        [已修复] app/models/user.rb:12——缺失空检查
        在下转换邮件前。
        → 已修复在提交 abc1234。自动回复。

        [虚假] lib/auth.rb:88——"令牌比较应该
        使用恒定时间比较。"我们已经使用
        此处 ActiveSupport::SecurityUtils.secure_compare。

        在 payment_service.rb 中修复竞争条件...

        认证令牌评论是虚假。
        选项：A）回复 Greptile 解释  B）反正修复  C）忽视

你：    A

Claude: 回复 Greptile。所有测试通过。
        PR：github.com/you/app/pull/42
```

三个 Greptile 评论。一个真实修复。一个自动确认。一个虚假推回带回复。总额外时间：大约 30 秒。
