# 每周工程回顾

生成分析提交历史、工作模式和代码质量指标的
综合工程回顾。团队感知：识别运行命令的用户，
然后分析每个贡献者，具有每人赞扬和成长机会。

## 参数

- 默认：最后7天
- `24h`：最后24小时
- `14d`：最后14天
- `30d`：最后30天
- `compare`：比较当前窗口与先前相同长度窗口

## 第1步：收集原始数据

```bash
git fetch origin main --quiet
git config user.name
git config user.email
```

由 `git config user.name` 返回的名字是**"你"**...
正在读这个回顾的人。所有其他作者都是队友。

运行所有这些git命令：

```bash
# 所有提交带时间戳、主题、哈希、作者、改变的文件
git log origin/main --since="<window>" --format="%H|%aN|%ae|%ai|%s" --shortstat

# 每提交测试与总LOC分解带作者
git log origin/main --since="<window>" --format="COMMIT:%H|%aN" --numstat

# 提交时间戳用于会话检测和每小时分布
git log origin/main --since="<window>" --format="%at|%aN|%ai|%s" | sort -n

# 最常改变的文件（热点分析）
git log origin/main --since="<window>" --format="" --name-only | grep -v '^$' | sort | uniq -c | sort -rn

# 提交消息中的PR号
git log origin/main --since="<window>" --format="%s" | grep -oE '[#!][0-9]+' | sort -t'#' -k1 | uniq

# 每作者文件热点
git log origin/main --since="<window>" --format="AUTHOR:%aN" --name-only

# 每作者提交计数
git shortlog origin/main --since="<window>" -sn --no-merges

# 测试文件计数
find . -name '*.test.*' -o -name '*.spec.*' -o -name '*_test.*' -o -name '*_spec.*' 2>/dev/null | grep -v node_modules | wc -l

# 窗口中改变的测试文件
git log origin/main --since="<window>" --format="" --name-only | grep -E '\.(test|spec)\.' | sort -u | wc -l
```

## 第2步：计算指标

计算并呈现这些指标：

- **提交到main：** N
- **贡献者：** N
- **合并的PRs：** N
- **总插入：** N
- **总删除：** N
- **添加的净LOC：** N
- **测试LOC（插入）：** N
- **测试LOC比率：** N%
- **版本范围：** vX.Y.Z → vX.Y.Z
- **活跃天：** N
- **检测到的会话：** N
- **平均LOC/会话小时：** N

然后立即下方显示**每作者排行榜**：

```
贡献者              提交    +/-          顶部区域
你 (garry)          32     +2400/-300   browse/
alice               12     +800/-150    app/services/
bob                  3     +120/-40     tests/
```

按提交数量降序排列。当前用户总是首先出现，标记为"你（名称）"。

## 第3步：提交时间分布

在本地时间显示每小时直方图：

```
小时   提交   ████████████████
 00:    4      ████
 07:    5      █████
 ...
```

识别：
- 峰值小时
- 死亡区域
- 双峰模式（早上/晚上）对比连续
- 深夜编码集群（晚上10点后）

## 第4步：工作会话检测

使用**45分钟间隙**阈值检测连续提交之间的会话。

分类会话：
- **深度会话**（50+分钟）
- **中等会话**（20-50分钟）
- **微型会话**（<20分钟，单提交）

计算：
- 总活跃编码时间
- 平均会话长度
- 每小时活跃时间的LOC

## 第5步：提交类型分解

按传统提交前缀（feat/fix/refactor/test/chore/docs）分类。
显示为百分比栏：

```
feat:     20  (40%)  ████████████████████
fix:      27  (54%)  ███████████████████████████
refactor:  2  ( 4%)  ██
```

如果修复比率超过50%，则标记... 信号一个"快速发货，快速修复"
模式，可能表示审查空白。

## 第6步：热点分析

显示10个最常改变的文件。标记：
- 改变5+次的文件（流失热点）
- 热点列表中的测试文件与生产文件
- VERSION/CHANGELOG频率

## 第7步：PR大小分布

估计PR大小并将其分类：
- **小**（<100 LOC）
- **中**（100-500 LOC）
- **大**（500-1500 LOC）
- **超大**（1500+ LOC）

## 第8步：焦点得分+ 本周发货

**焦点得分：** 接触单个最常改变的顶级目录的提交百分比。
更高=更深入的专注工作。更低=分散的上下文切换。

**本周发货：** 窗口中的单个最高LOC PR。
突出显示PR号、改变的LOC和为什么它很重要。

## 第9步：团队成员分析

对于每个贡献者（包括当前用户），计算：

1. **提交和LOC** ... 总提交、插入、删除、净LOC
2. **焦点区域** ... 他们接触最多的目录/文件（前3个）
3. **提交类型组合** ... 他们个人的feat/fix/refactor/test分解
4. **会话模式** ... 他们何时编码（峰值小时）、会话计数
5. **测试纪律** ... 他们个人的测试LOC比率
6. **最大发货** ... 他们的单个最高影响提交或PR

**对于当前用户（"你"）：** 最深入的处理。
包括所有会话分析、时间模式、焦点得分。以第一人称写框架。

**对于每个队友：** 2-3句话覆盖他们发货的内容和他们的模式。
然后：

- **赞扬**（1-2个具体事物）：锚定在实际提交。
- **成长机会**（1个具体事物）：框架为级升，而不是批评。

## 第10步：周对周趋势（如果窗口>= 14天）

分成每周桶并显示趋势：
- 每周提交数（总和和每作者）
- 每周LOC
- 每周测试比率
- 每周修复比率
- 每周会话计数

## 第11步：条纹追踪

计算从今天开始至少有1个提交的连续天数：

```bash
# 团队条纹
git log origin/main --format="%ad" --date=format:"%Y-%m-%d" | sort -u

# 个人条纹
git log origin/main --author="<user_name>" --format="%ad" --date=format:"%Y-%m-%d" | sort -u
```

显示两者：
- "团队运送条纹：47个连续天"
- "你的运送条纹：32个连续天"

## 第12步：加载历史和比较

检查 `memory/` 中的先前回顾历史：

如果先前回顾存在，加载最新的一个并计算增量：

```
                    最后    现在        增量
测试比率：          22%  →  41%         ↑19pp
会话：              10   →  14          ↑4
LOC/小时：         200   →  350         ↑75%
修复比率：          54%  →  30%         ↓24pp（改进）
```

如果没有先前回顾存在，注意"首次记录回顾，下周再运行以查看趋势。"

## 第13步：保存回顾历史

将JSON快照保存到 `memory/retro-YYYY-MM-DD.json`
带指标、作者、版本范围、条纹和可推文摘要。

## 第14步：编写叙述

**格式用于Telegram**（要点、粗体、最终输出中无markdown表）。

结构：

**可推文摘要**（第一行）：
> 3月1周：47次提交（3个贡献者），3.2k LOC，38%测试，12个PRs，峰值：晚上10点 | 条纹：47天

然后是各部分...

## 重要规则

- **所有时间以本地时区。** 永远不设置 `TZ`。
- **格式用于Telegram。** 使用要点和粗体。避免markdown表。
- **赞扬锚定在提交中。** 永远不说"伟大的工作"，而不命名什么是好的。
- **成长领域锚定在数据中。** 永远不批评而不有证据。
- **保存历史。** 每个回顾保存到 `memory/` 以用于趋势追踪。
