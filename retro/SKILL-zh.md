# /retro: 周工程回顾 — 提交分析 + 团队模式

你是周工程回顾的主持人。拉取所有这周写的代码，分析提交，计算代码质量指标，和报告团队进度 — 谁在船舶，谁在 blocked，代码质量趋势是什么？

## 使用方式

**单个贡献者：** `/retro` — 分析自上周以来的你的提交并生成一份摘要

**跨团队：** `/retro --team` — 分析所有贡献者的提交，为每一个生成粒度统计

**对比到基线：** `/retro --baseline main` — 把分析约束在特性分支上（相对于 main）vs 所有时间

## 原则

一个好的回顾回答了四个问题：

1. **谁交付了什么？** 所有提交、更改大小、受影响的代码区域
2. **代码质量怎样？** 复杂性、测试覆盖率、代码评论趋势
3. **团队是否 blocked？** 谁完成、谁卡住、什么文件是瓶颈
4. **我们学了什么？** 哪些模式出现了？什么改进了什么变差了？

## 工作流

### 阶段 1: 参数解析

```bash
SINCE="1 week ago"
MODE="individual"  # or "team" or "team-detailed"
BASELINE="origin/main"
REPO="."
```

**覆盖选项：**
- `--since "2 weeks ago"` → 回溯 2 周
- `--team` → 跨贡献者分组统计
- `--baseline <branch>` → 仅特性分支相对于基线
- `--repo <path>` → 分析特定 repo（多 repo 设置）

### 阶段 2: 提交收获

```bash
git log --since "$SINCE" --format="%H %an %ae %aI %s" --name-status
```

对每个提交捕获：
- SHA
- 作者名和邮箱
- 日期/时间
- 消息主题
- 更改的文件列表（A/M/D）

**跨多个贡献者时，按作者聚合。**

### 阶段 3: 指标计算

对每个提交计算：

| 指标 | 计算 | 为什么 |
|---------|-----------|---------|
| LSoC | 变化的行 = 添加 + 删除 | 速度和风险。大改 = 更难审查。 |
| 文件 | 改动文件总数 | 范围。多个文件 = 更广泛的影响。 |
| 复杂性 | (LSoC / 文件) | 平均 delta 每个文件。高 = 密集、高风险提交。 |
| 区域 | 涉及的顶级目录 | 关注点分离。许多区域 = 混合关注点。 |
| 关键字 | 消息匹配："fix", "feat", "refactor", "test", "docs" | 工作类型的分布。 |
| 测试？| "test" 或 "+test" 或测试文件改动 | 是否提交包括测试。测试 > 代码 = 高信心。 |

### 阶段 4: 焦点分析

**什么文件最常改动？**

```bash
git log --since "$SINCE" --name-only --format="" | sort | uniq -c | sort -rn | head -20
```

这些是**热点文件**。高变化 = 高维护风险。

```markdown
### 热点文件（高变化且必须审查）
- src/database.ts: 47 改动（12 人，3 地点）
- src/api.ts: 34 改动（8 人）
- ...
```

**哪些文件有冲突？** 查看多个人在同一个文件中的改动：

```bash
git log --since "$SINCE" --format="%an" --name-only -- src/**
```

哪些作者触及同一个文件？那些文件是**集成点**。如果它们频繁改动，集成风险很高。

### 阶段 5: 质量评估

**代码回顾覆盖** — 多少提交有评论？（使用 git log --all --grep="Reviewed-by" 或检查 PR 历史）

**测试交付率** — 有多少提交触及测试文件？

```
提交数: 45
有测试的: 12
测试交付率: 26%
```

低 (<50%) 应该标记为 "需要关注"。

**提交细粒度** — 平均改动大小

```
平均 LSoC: 187 行
平均文件: 4 个文件
平均复杂性: 47 LSoC/文件

范围：最小 5 行（1 文件） → 最大 1,847 行（23 文件）
```

过大的平均值（>500 行）提示提交可能没有足够细粒度审查。

**逻辑一致性** — 提交消息是否清晰表达意图？

```bash
git log --since "$SINCE" --format="%s" | grep -i "wip\|tmp\|debug\|fix typo\|lol"
```

任何临时消息应该被标记为审查候选。

### 阶段 6: 团队视图（`--team` 模式）

**对每个贡献者生成：**

```markdown
## 贡献者：Bob

### 交付
- 提交：8
- 更改：1,247 行
- 文件：31

### 聚焦
- 主要区域：src/database, src/api
- 热点文件：migrations/001_users_table.sql (3 改动)

### 质量
- 测试交付率：50%（4/8 提交有测试）
- 平均提交大小：156 行
- 代码回顾：6/8 有评论

### 风险
- 最大改动：487 行 （1 个文件）— SHA: abc1234
- 高复杂性提交：2 （平均 89 LSoC/文件）

### 情感
- 消息基调：[professional, experimental, urgent, thorough]
- 工作模式：[steady, burst, blocked]
```

**基调检测：** 扫描消息内容和频率

- "fix fix fix 修复" → 可能在灭火
- "refactor ... refactor ... 重构" → 正在清理技术债
- "feat ... feat ... feat" → 新特性爆发
- "wip ... wip ... wip" → 可能 blocked

**工作模式检测：** 时间戳模式

- 许多提交在同一天 vs 分散到全周 → "burst" 对 "steady"
- 周末提交 → "加班" 风险

标记任何 "burst" 或 "加班" 进度为需要关注。

### 阶段 7: 健康指数

计算整体团队**代码健康分数** (0-100)：

```
基础：100

传扣：
- 低测试交付率 (<40%)：        -15
- 高平均 LSoC (>400):           -10
- 高热点文件变化 (>20 改动):    -10
- 许多评论缺失提交 (<60%):      -5
- 临时消息发现 (wip/tmp/debug): -8
- 团队阻滞信号 (blocked 分支):  -5

分数 = 100 - (传扣之和)
```

示例：
```
基础：100
- 低测试交付（35%）：-15
- 高平均提交大小（523 行）：-10
- 2 个临时消息：-8
- 3 文件有 (>20 改动)：-10

分数：100 - 43 = 57 / 100 ("关注需要" 中等)
```

**分数等级：**
- 80-100: 健康 ✓
- 60-79: 关注需要 ⚠️
- <60: 高风险 🚨

### 阶段 8: 趋势分析

**对比到上周：** 如果持久化了先前的报告，计算：

```
这周：78 分
上周：82 分
Δ：-4

原因可能：
- 测试交付下降 50% → 45%
- 平均提交大小上升 180 → 240 行
- 一个贡献者的大 PR（487 行）
```

标记任何负 Δ > 5 分为需要调查的。

### 阶段 9: 阻滞检测

寻找"卡住"的贡献者：

- 在同一个文件中反复 cherry-pick/revert？
- 提交消息包含 "conflict", "merge", "rebase"？
- PR 打开超过 5 天，没有合并？
- 没有提交在过去 3 天？

```markdown
### 可能被 Blocked 的贡献者
- Alice: PR #234 开放 8 天（等待回顾）
- Bob: 在 src/database.ts 最后 4 个提交有冲突信号
```

提议帮助（`/investigate` 对 merge 冲突或回顾延迟）。

### 阶段 10: 学习（当 --team 时）

**谁引入了最多的 bug？** （相关提交随后从 revert 搜索）

```bash
git log --all --grep="Revert" --format="%s" | grep -oP 'Revert "\K[^"]*'
```

追踪被还原的提交回他们的作者。不是责备 — 是学习。

**谁的 PR 有最多回合的评论？** （ PR 历史）

这表示沟通或期望对齐中的潜在问题。

### 阶段 11: 生成报告

**输出格式：** Markdown 到 `.gstack/retro-{date}.md`（个人）或 `~/.gstack/projects/{slug}/retro-{date}-team.md`（团队）

**结构：**

```markdown
# 周工程回顾：{日期范围}

## 快照
- 团队大小：N 人
- 提交数：M
- 更改行数：X
- 测试交付率：Y%
- 健康分数：Z / 100

## 贡献者摘要

（如果 --team：表格所有贡献者）

| 贡献者 | 提交 | LSoC | 测试 | 评论 | 优先事项 |
|------------|-------|-------|---------|----------|----------|
| Alice | 12 | 1,247 | 50% | 10/12 | 稳定 |
| Bob | 8 | 1,847 | 25% | 4/8 | ⚠️ 低测试 |

## 热点文件
[文件名，计数，人员，风险]

## 质量指标
[LSoC, 复杂性, 测试, 审查]

## 工作模式 & 情感
[按贡献者的模式检测]

## 风险
[任何 "关注需要" 或 "高风险" 分数]

## 学习
[趋势, 被阻滞的贡献者, 潜在改进]

## 建议
[基于指标的下周改进]
```

### 阶段 12: 持久化和对比

**保存报告和指标到：**

本地：`.gstack/retro/{date}-{mode}.json`
全局：`~/.gstack/projects/{slug}/retro-{date}-{mode}.json`

**JSON 格式：** 原始指标（以便下周对比）
```json
{
  "date": "2024-01-15",
  "since": "2024-01-08",
  "mode": "team",
  "contributors": {
    "alice": {
      "commits": 12,
      "lsoc": 1247,
      "test_rate": 0.50,
      "comment_rate": 0.83
    }
  },
  "health_score": 78,
  "hotspot_files": [...]
}
```

## 规则

1. **每周运行**（周五或周一）一致
2. **永不跳过个体评估** — 即使从来没有 team 旗标运行，也永不跳过它，每个月一次交叉成员一致
3. **永不发表责备** — 报告是学习，不是评估绩效
4. **永远显示数据，然后解释** — 显示数字，然后关于含义的叙述
5. **同步团队贡献者列表** — `/retro --team` 应该包括所有人谁在 repo 的 git 配置中，使用 `git log --format=%an | sort -u`
