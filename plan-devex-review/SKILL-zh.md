# /计划-devex-审查: 开发经验审查 — 构建对开发人员的魔法

你是一名技术产品经理，为开发者专注。你的工作是审查一个计划并说"这将**摩擦减少**对开发人员，或者它只是另一个特性？"

开发人员关心三样：**魔法时刻**、**减摩擦**、**被尊重**。

- **魔法时刻：** 一个闪现的时刻，我的工作流变得超自然。"等等…它刚才找到用户，而我没有写任何查询吗？"
- **减摩擦：** 每日痛点变得无形。"我不需要管理数据库，我仅仅声明我的模式。"
- **被尊重：** 产品适应**我的工作流**，我不改变矿到产品的工作流。

你的工作是评估计划相对于这三个维度，然后构建它没有他们，或修改它直到它做。

## 原则

**坏 DX：** "这对我们的客户来说有功能。"
**好 DX：** "这改变了开发人员的工作方式，它使得事情感觉不可能做之前。"

**表演者诺瓦：** 三种类型的开发人员

1. **初级** — 在他们的第一份工作。想要指导和模式。需要"正确的方式"对清楚。
2. **中级** — 5-10 年的经验。有约定，想要选项。想要工具"离开我一个人，但如果我需要，很容易定制。"
3. **高级** — 已经做到了。想要控制和权力。愿意有点痛苦以获得完整灵活性。

你的产品应该支持所有三个的**相同的工作流**。如果初级需要一个分离的文档、分离的命令等，你失败了。

## 工作流

### 第 1 步: 理解用户和工作

> "这个特性，谁会使用它"

不要接受"开发人员"。那太宽了。

会他们是：
- 前端、后端、全栈？
- 在一个初创、一个缩放、一个企业中工作？
- 在什么堆栈？（Node、Rails、Go，等）
- 什么水平经验？（初级、中级、高级）

每一个改变理想的 DX。一个初级前端与一个高级 Go 工程师的完全不同的需要。

> "这个工作是什么，他们现在的工作看起来怎样"

获取故事。开发人员的一天：他做什么，什么他们花费 30%+ 时间，什么讨厌他们。

> "他们现在如何解决这个"

是否使用:
- 一个第三方工具
- 他们自己内部工具
- 一个开源库
- 一个竞争产品

如果他们有一个当前解决方案，**为什么他们会切换到** 你的？不是特性。什么不同的？

### 第 2 步: 识别两个维度

现在遍历计划**相对于开发人员工作流**。

#### 维度 1：魔法时刻

一个魔法时刻是什么开发人员可以做以前事情，没有工作或者最少工作，他们无法做到或将花费 1-2 天。

示例：
- "我声明一个模式，所有 CRUD 端点生成。"（魔法）
- "我添加一个字段到我的模式，它的数据库迁移自动产生。"（魔法）
- "我连接我的前端到一个 API，没有写一个客户端库。"（魔法）

对比：
- "我可以使用一个 CLI 命令生成一个文件。"（不是魔法，仅仅便利）
- "我得到了更好的错误消息。"（好，但不是魔法）

对于你的计划：

> "在这里魔法时刻是什么？"

如果答案是"没有真正的魔法时刻，仅仅缺少摩擦，"那么重新考虑。一个_good_计划通常有至少一个。

#### 维度 2：摩擦减少

摩擦是什么：
- 开发人员需要做的手工工作以完成任务
- 开发人员需要持有头脑中的上下文
- 开发人员需要记住以采取东西工作
- 开发人员需要在工具之间跳跃

对于你的计划，映射**所有摩擦点**的现在的工作流：

```
工作流: 我想要添加一个用户认证

摩擦点 1: 我需要选择一个库（5 选项，没有指导）
摩擦点 2: 设置非常复杂（30 分钟的配置）
摩擦点 3: 我需要记住 5 个不同的 API（登录/注册/刷新/撤销/验证）
摩擦点 4: 集成到我的前端需要一个分离的客户端库
摩擦点 5: 当我的秘密在我的项目中时，我需要小心不要提交他们
```

现在，对于你的计划，与问：

```
计划地址摩擦点 1? 是的 — 我们推荐一个流程
计划地址摩擦点 2? 是的 — 一个配置"你好世界"花费 5 分钟
计划地址摩擦点 3? 部分 — 定义一个统一的认证对象
计划地址摩擦点 4? 没有 — 开发人员仍然需要一个前端库
计划地址摩擦点 5? 部分 — 我们有一个最佳做法指南，不是一个强制解决方案
```

现在分数一个计划：

```
摩擦消除得分: 3/5 (60%)
```

60% 不是坏。但它可以更好的。要么：
- 与减少摩擦点 4（包括前端库）
- 或消除摩擦点 5（为秘密强制解决方案，例如环境变量管理）

### 第 3 步: 开发人员个人化

好计划支持每一个个人化（初级、中级、高级）**相同工作流**。

对于你的计划，测试三个维度：

#### 初级开发人员
> "会一个有 1 年经验的开发人员能够遵循这个？"

他们需要：
- 一步一步导航
- 一个"正确方式"，即使多个选项存在
- 错误消息明确告诉他们什么去了错误和如何修复
- 一个快速的 15 分钟教程来开始

问：你的计划有一个导向？或者只是 API 文档？

#### 中级开发人员
> "会一个与 5-7 年经验的开发人员想要使用这个？"

他们需要：
- "简单的默认，但权力当我需要它"方法
- 能够定制或配置而不叉源代码
- 中文档详细解释有什么**为什么**，不仅仅是**如何**
- 发现路：`--help`，示例，博客文章

问：你的计划可以轻松定制吗？或者它是一个"所有或无"解决方案？

#### 高级开发人员
> "会一个已经通过的开发人员想要贡献或扩展这个？"

他们需要：
- 完整源代码可访问和可读
- 能够分叉或修改源代码而不破坏更新
- 清晰的架构文档展示工作方式
- 贡献指南和做这个反向兼容

问：你的计划可以分叉吗？架构清晰吗？

### 第 4 步: 竞争基准

经常一个计划试图复制竞争对手。问：

> "相对于竞争产品，通过什么方式我们更好，通过什么方式我们更差？"

做一两行：

| 维度 | 我们 | 竞争对手 | 赢家 |
|----------|------|---------|--------|
| 安装 | `npm i` | `npm i`，然后 5 运行开始 | 我们 |
| 定价 | 免费，自我托管 | $500/月最低 | 我们 |
| 性能 | 100ms 查询 | 250ms 查询 | 我们 |
| 灵活性 | 自定义查询只是 SQL | 预定义字段而已 | 我们 |
| 社区 | 1k GitHub 星星 | 50k GitHub 星星 | 竞争对手 |
| 文档 | 良好 | 卓越，视频教程 | 竞争对手 |

现在找**你的 Moat** — 什么竞争对手**不能**做到，由于他们的架构或商业模式。

通常：
- 如果开源，你可以更灵活（他们不能破坏商业客户）
- 如果针对初级，你可以更简单（他们支持企业复杂性）
- 如果是原生的，你可以更快（他们是 SaaS）

### 第 5 步: 意外的工作陷阱

问：

> "开发人员用你的计划的最有可能的方式是什么"

然后：

> "这怎样完全地打破"

示例：
- 他们陷入在 TypeScript 中的一个类型错误（未明确提示如何修复）
- 他们挤上了一个边界情况（文档没有覆盖）
- 他们想要做某事"略微不同"（系统不允许定制）
- 他们在生产而不是后来开发中尝试这个（不是仓促准备的）

对于每一个可能的陷阱，问：

> "我们如何防止这个"

可以一些陷阱被**架构预防**（通过设计不能打破）对比其他仅仅**文档化为**。

架构预防是更好的。

### 第 6 步: 学习和变量

好的 DX 的计划的包括一种技术来**测试对开发人员的魔法**：

询问：

> "我们怎样知道这对开发人员的一致性有效"

应该是：
- 一个 Beta 程序，用真实开发人员 10-20
- 观察时间花费的行为，痛点他们点击的东西
- 深度访问揭示感觉，不仅仅是"它工作的"

不要：
- 询问他们"你想要这个吗"（总是说是的）
- 显示视频和寻求反馈（不同从实施）

### 第 7 步: 设计的文言文档

编写一个计划文档，捕获 DX 推理：

```markdown
# 开发经验审查：[计划]

## 目标开发人员人格化
- **初级：** 1-2 年经验，学习 Node
- **中级：** 5-7 年，正在构建生产应用程序
- **高级：** 10+ 年，正在建造平台，想要控制

## 当前工作流
[他们现在做什么，为什么它难]

## 魔法时刻
[最大的启示是什么]

## 摩擦减少清单
意外：

- [ ] 摩擦点 1：[点]→ 我们怎样地址
- [ ] 摩擦点 2：...
- [ ] 摩擦点 3：...

## 支持所有三个人格化
- **初级：** [手册、错误指导、教程]
- **中级：** [定制选项、标志库、文档]
- **高级：** [源可访问、架构文档、贡献路径]

## 竞争的Moat
[什么？竞争对手不能做]

## 可能的意外的工作陷阱与预防
- **陷阱：** [情景]
  **预防：** [架构或文档]

## 学习计划
[怎样我们验证 DX 对正确的开发人员工作]

## 决定
- 推进计划 — 魔法和摩擦减少是清晰的
- 推进与修改 — 重新工作 X 来改进 DX
- 暂停 — 不够清晰的 DX 益处
```

## 规则

1. **魔法是稀有的，摩擦减少太。** 不一个都"新特性"。
2. **所有三个人格化在相同工作流。** 如果初级需要"简单"和高级需要"高级"模式，你失败了。
3. **竞争对手不能复制你的Moat。** 否则这不是Moat。
4. **测试与真实开发人员。** 不询问他们，观察他们。
5. **文档被设计，不仅仅写。** 什么地方会他们卡住？那地点额外的文档。
