# 登陆前审查清单

## 说明

审查 `git diff origin/main` 输出中列出的问题。具体说明 — 引用`file:line`并建议修复。跳过任何没问题的东西。仅标记实际问题。

**两遍审查：**
- **第一遍（关键）：** 首先运行SQL和数据安全、竞态条件、LLM输出信任边界、Shell注入和枚举完整性。最高严重性。
- **第二遍（信息性）：** 运行下面剩余的类别。较低严重性但仍然需要处理。
- **专家类别（由平行子智能体处理，不是此清单）：** 测试空隙、死代码、魔术数字、条件副作用、性能和包大小影响、密码学和熵。请参阅 `review/specialists/` 了解这些。

所有发现通过修复优先审查进行处理：明显的机械修复自动应用，
真正模糊的问题被整理成单个用户问题。

**输出格式：**

```
登陆前审查：N问题（X关键，Y信息性）

**自动修复：**
- [file:line] 问题 → 已应用修复

**需要输入：**
- [file:line] 问题描述
  建议修复：建议的修复
```

如果未发现问题：`登陆前审查：未发现问题。`

简洁。对于每个问题：一行描述问题，一行修复。无导言，无摘要，无"总体看起来不错"。

## 审查类别

### 第一遍 — 关键

#### SQL和数据安全
- SQL中的字符串插值（即使值是 `.to_i`/`.to_f` — 使用参数化查询（Rails: sanitize_sql_array/Arel; Node: prepared statements; Python: parameterized queries））
- TOCTOU竞齿：检查然后设置应该是原子的 `WHERE` + `update_all` 模式
- 绕过模型验证进行直接数据库写入（Rails: update_column; Django: QuerySet.update(); Prisma: raw queries）
- N+1查询：缺少关联的急加载（Rails: .includes(); SQLAlchemy: joinedload(); Prisma: include）

#### 竞态条件和并发
- 读-检查-写没有唯一性约束或捕获重复键错误并重试（例如，`where(hash:).first` 然后 `save!` 不处理并发插入）
- 找或创建没有唯一数据库索引 — 并发调用可以创建重复
- 不使用原子 `WHERE old_status = ? UPDATE SET new_status` 的状态转换 — 并发更新可以跳过或重复应用转换
- 不安全的HTML渲染（Rails: .html_safe/raw(); React: dangerouslySetInnerHTML; Vue: v-html; Django: |safe/mark_safe）基于用户控制的数据（XSS）

#### LLM输出信任边界
- LLM生成的值（电子邮件、URL、名称）写入数据库或传递给邮件发送者，不进行格式验证。在持久化前添加轻量级保护（`EMAIL_REGEXP`、`URI.parse`、`.strip`）。
- 结构化工具输出（数组、哈希）在数据库写入前接受，无类型/形状检查。
- LLM生成的URL获取，无允许列表 — SSRF风险（Python: `urllib.parse.urlparse` → 在 `requests.get`/`httpx.get` 前根据黑名单检查主机名）
- LLM输出存储在知识库或矢量DB中，无清理 — 存储的提示注入风险

#### Shell注入（Python特定）
- `subprocess.run()` / `subprocess.call()` / `subprocess.Popen()` 与 `shell=True` 和命令字符串中的f字符串/`.format()` 插值 — 改用参数数组
- `os.system()` 与变量插值 — 用参数数组替换为 `subprocess.run()`
- 对LLM生成代码进行 `eval()` / `exec()` 而不沙盒

#### 枚举和值完整性
当diff引入新枚举值、状态字符串、层级名称或类型常数时：
- **追踪它通过每个消费者。** 读（不仅grep — 读）该值的每个文件。如果任何消费者不处理新值，标记它。常见漏掉：添加值到前端下拉列表但后端模型/计算方法没有持久化它。
- **检查允许列表/过程数组。** 搜索包含兄弟值的数组或`%w[]`列表（例如，如果添加"修订"到层级，找到每个 `%w[quick lfg mega]` 并验证"修订"包含在需要的地方）。
- **检查`case`/`if-elsif`链。** 如果现有代码在枚举上分支，新值是否下降到错误的默认值？
为此：使用Grep找到兄弟值的所有引用（例如grep找"lfg"或"mega"找到所有层级消费者）。读每个匹配。这步需要读diff外的代码。

### 第二遍 — 信息性

#### 异步/同步混合（Python特定）
- 同步 `subprocess.run()`、`open()`、`requests.get()`在`async def`端点内 — 阻塞事件循环。改用 `asyncio.to_thread()`、`aiofiles` 或 `httpx.AsyncClient`。
- 异步函数内的 `time.sleep()` — 使用 `asyncio.sleep()`
- 不使用 `run_in_executor()` 包装的异步上下文中的同步数据库调用

#### 列/字段名称安全
- 在ORM查询中验证列名（`.select()`、`.eq()`、`.gte()`、`.order()`）对照实际数据库架构 — 错误的列名静默返回空结果或抛出吞咽的错误
- 检查`.get()`调用在查询结果上使用实际选择的列名
- 与架构文档交叉参考

#### 死代码和一致性（仅版本/更新日志 — 其他项由可维护性专家处理）
- PR标题和VERSION/CHANGELOG文件之间的版本不匹配
- 不准确描述变化的CHANGELOG条目（例如，"从X改为Y"当X从不存在）

#### LLM提示问题
- 提示中的0索引列表（LLM可靠地返回1索引）
- 列出可用工具/能力的提示文本，不匹配实际在 `tool_classes`/`tools` 数组中连接的
- 在多个地方指定的词/令牌限制，可能漂移

#### 完整性空隙
- 快捷实现，其中完整版本成本<30分钟CC时间（例如，部分枚举处理、不完整的错误路径、缺少直接添加的边缘情况）
- 仅显示人类团队工作量估计的选项 — 应显示人类和CC+gstack时间
- 测试覆盖的空隙，其中添加缺失的测试是"湖"而不是"海"（例如，缺失的负路径测试、缺失的镜像快乐路径结构的边缘情况测试）
- 功能实现在80-90%当100%可实现为适度额外代码

#### 时间窗口安全
- 假设"今天"覆盖24小时的日期键查找 — PT上午8点报告仅看到午夜→上午8点在今天的键下
- 相关功能之间的时间窗口不匹配 — 一个使用每小时桶，另一个使用每日键用于相同数据

#### 类型强制在边界处
- 穿过Ruby→JSON→JS边界的值，其中类型可能变化（数字vs字符串） — 哈希/摘要输入必须规范化类型
- 不调用`.to_s`或等价物在序列化前的哈希/摘要输入 — `{ cores: 8 }` vs `{ cores: "8" }` 生成不同的哈希

#### 视图/前端
- 部分中的内联 `<style>` 块（每次渲染重新解析）
- 视图中的O(n*m)查找（循环中的 `Array#find` 而不是 `index_by` 哈希）
- Ruby端的 `.select{}` 过滤数据库结果，这可能是 `WHERE` 子句（除非有意避免前导通配符 `LIKE`）

#### 分布和CI/CD管道
- CI/CD工作流变化（`.github/workflows/`）：验证构建工具版本匹配项目要求、硬件名称/路径正确、密钥使用 `${{ secrets.X }}` 不硬编码值
- 新工件类型（CLI二进制、库、包）：验证发布/发布工作流存在并针对正确的平台
- 跨平台构建：验证CI矩阵覆盖所有目标OS/arch组合，或文档哪些未测试
- 版本标签格式一致性：`v1.2.3` vs `1.2.3` — 必须跨VERSION文件、git标签和发布脚本匹配
- 发布步骤幂等性：重新运行发布工作流不应失败（例如，`gh release delete` 在 `gh release create` 前）

**不要标记：**
- 带有现有自动部署管道的Web服务（Docker构建+ K8s部署）
- 团队外未分布的内部工具
- 仅测试CI变化（添加测试步骤，不发布步骤）

## 修复优先启发式

此启发式由 `/review` 和 `/ship` 引用。它确定智能体是否自动修复发现或询问用户。

```
自动修复（智能体不询问地修复）：   询问（需要人工判断）：
├─ 死代码/未使用的变量            ├─ 安全（身份验证、XSS、注入）
├─ N+1查询（缺失急加载）          ├─ 竞态条件
├─ 陈旧的注释与代码矛盾          ├─ 设计决定
├─ 魔术数字→命名常数              ├─ 大型修复(>20行)
├─ 缺失LLM输出验证                ├─ 枚举完整性
├─ 版本/路径不匹配                ├─ 移除功能
├─ 分配但从不读的变量             └─ 任何改变用户可见的
└─ 内联样式、O(n*m)视图查找         行为
```

**经验法则：** 如果修复是机械的，高级工程师会在没有讨论的情况下应用，那就是AUTO-FIX。如果合理的工程师可能对修复意见不一致，那就是ASK。

**关键发现默认倾向于ASK** （本质上风险更高）。
**信息性发现默认倾向于AUTO-FIX** （更机械）。

## 抑制 — 不要标记这些

- "X与Y冗余"当冗余是无害的并有助于可读性（例如，`present?` 与 `length > 20` 冗余）
- "添加注释解释为什么选择了此阈值/常数" — 阈值在调整期间改变，注释会变陈旧
- "此断言可以更严格"当断言已覆盖行为
- 建议仅一致性的变化（包装一个值在条件中以匹配另一个常数如何被保护）
- "正则表达式不处理边缘情况X"当输入受限且X在实践中从不发生
- "测试同时执行多个保护" — 没关系，测试不需要隔离每个保护
- 评估阈值变化（max_actionable、最小分数） — 这些经验调整并不断变化
- 无害的无操作（例如，`.reject` 在一个永不在数组中的元素上）
- 你正在审查的diff中已经处理的任何东西 — 评论前阅读完整的diff
