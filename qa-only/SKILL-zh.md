# /qa-only: QA 报告（没有修复）

你是一名 QA 工程师。像真正的用户一样测试应用程序 — 点击每件事，填写每个表单，检查每个状态。记录你发现的每个问题。**但不要修复任何东西。** 仅仅报告。

这对以下情况是：
- 你想要证据一个功能运行之前发布
- 第三方审查需要一个独立的 QA 验证
- 你想要 bug 报告的一个列表，期后有其他人修复它们

## 快速启动

```bash
/qa-only <url>
```

可选参数：
- `--quick`：30 秒烟雾测试（主页 + 前 5 个链接）
- `--exhaustive`：完整应用 QA，每一个页面，每一个交互状态
- 默认：标准 QA（主要页面 + 常见交互）

## 工作流（报告-only）

### 阶段 1: 初始化

```bash
$B status 2>/dev/null | grep -q "Mode: cdp" && echo "CDP_MODE=true" || echo "CDP_MODE=false"
```

检查先决条件：
- 确定目标 URL
- 查找浏览器二进制
- 创建输出目录 `.gstack/qa-reports/`

### 阶段 2: 导航应用

访问所有可达页面。对每一页：

```bash
$B goto <page-url>
$B screenshot -i -a -o "report/page-name.png"
$B console --errors
```

记录：
- 页面标题和 URL
- 视觉上是什么样子的（分类：主页、表单页、结果页等）
- 任何控制台错误或警告

### 阶段 3: 对每一页测试交互

不要要关闭和跳过。对每一页，逐行通过一个检查清单：

#### 链接
- [ ] 所有链接可点击？（没有死链接）
- [ ] 所有链接去正确的地方？（检查导航）
- [ ] 链接 `<a>` 标签还是一个 `<button>`？（应该是 `<a>` 对真链接）

#### 按钮
- [ ] 所有按钮可点击？
- [ ] 按钮禁用，当它们应该？（加载时）
- [ ] 按钮点击做什么他们说？（送出表单，导航，打开模式等）
- [ ] 有任何明显的按钮什么错过？（一个用户想要点击的）

#### 表单
- [ ] 所有字段可填充？
- [ ] 正式提交吗？
- [ ] 验证工作吗？（错误消息清晰吗？）
- [ ] 表单的成功状态做什么？（重新定向？消息？）
- [ ] 所有输入类型正确？（EMAIL 输入为电子邮件，数字输入为数字等）

#### 数据和内容
- [ ] 页面加载数据正确？
- [ ] 任何拼写错误或 AI Slop（奇怪坏词断行、缺失标点等）？
- [ ] 图像加载？（没有损坏图像位）
- [ ] 文本可读？（足够大，足够对比）

#### 错误处理
- [ ] 如果一个 API 是缓慢，什么发生？（加载旋转器？超时错误？冻结？）
- [ ] 如果一个 API 失败，什么显示？（清晰错误消息？重试按钮？）
- [ ] 如果一个用户离开一个字段空白，什么发生？（验证消息）

#### 响应性
- [ ] 页面在移动视口重新排列吗？（不水平溢出）
- [ ] 在平板视口看得好吗？
- [ ] 桌面视口看得好吗？

#### 内存泄漏和性能
- [ ] 页面重新加载吗？（应该非常快）
- [ ] 如果你多次浏览页面，速度变慢吗？（可能内存泄漏）
- [ ] 控制台有任何即将废弃警告？

### 阶段 4: 立即记录每个问题

**不要批处理。不要推迟。直接记录。**

为每一个问题：

1. 拍摄**证据屏幕截图** （问题的一个清晰的图像或之前/之后的对）
2. 写清晰的重现步骤：
   ```
   1. 去往 https://...
   2. 点击[按钮名]
   3. 填充表格 [这样和那样]
   4. ...
   结果：[发生什么，应该发生什么]
   ```
3. 分布**严重性**：
   - **关键**：应用是中断的，用户不能完成一个流
   - **高**：一个明显的错误，大多数用户会看到
   - **中等**：一个不一致或对一些用户的错误
   - **低**：一个微妙问题，边界情况，或仅仅化妆
4. 包括**框架线索**（后续修复者可能需要它）：
   - React？Vue？Next.js？Rails？Django？
   - 网络或服务器渲染？
   - 一个 SPA 还是多页应用？

### 阶段 5: 总结和计算得分

测试完成后，运行一个**健康检查**。

#### 健康分数标尺
```
基础：100

传扣：
- 关键问题（块用户）：          -15 每一个
- 高问题（明显错误）：           -5 每一个
- 中等问题（不一致）：           -2 每一个，最多 -10
- 低问题（边界情况、化妆）：     -1 每一个，最多 -5

分数 = 100 - (传扣)
```

示例：
```
基础：100
- 3 个关键问题（按钮不工作，表格不提交）：-45
- 2 个高问题（破损图像，打字错误）：-10
- 5 个中等问题（填充不一致）：-10
- 6 个低问题（UI 抛光）：最多 -5

分数：100 - 70 = 30 / 100 ("高风险" — 不准备生产)
```

#### 问题汇总
计算按严重性的计数：

```
关键：3
高：2
中等：5
低：6

总：16 个问题发现
```

### 阶段 7: 分配所有权和优先级

如果有一个 bug 追踪系统（GitHub, Jira, etc），创建 bug 票证：

- **关键问题：** 立即修复（P0）
- **高问题：** 修复之前发布（P1）
- **中等问题：** 修复下一个版本（P2）
- **低问题：** 修复时时间允许（P3）

## 规则

1. **不修复任何东西。** 你的工作是报告。修复来得以后。
2. **证据首先。** 每个问题 = 至少一个屏幕截图或视频。
3. **重现步骤必须列出。** 一个后期修复者应该能够完全重现仅应用你的步骤。
4. **框架线索帮助。** 告诉修复者你认为什么原因问题（React vs API vs 设计等）。
5. **健康分数是目标。** 目标是清晰的。一个报告与证据得分比一个写一个令人惊讶的。
6. **没有批评，仅仅事实。** 不"这个代码很烂。" 仅仅"这个链接不工作。"
7. **重现一切两次。** 再次验证每个问题，确保它**不是**用户错误（他们点击错误地方，等）。

## 对比 `/qa`

| 特征 | `/qa` | `/qa-only` |
|---------|--------|-----------|
| 报告任何问题 | ✓ | ✓ |
| 修复问题 | ✓ | ✗ |
| 原子提交 | ✓ | ✗ |
| 测试验证修复 | ✓ | ✗ |
| 时间承诺 | 高（包括修复） | 低（仅报告） |
| 最好用于 | 开发者想要完整 QA + 修复 | 第三方审查或 QA 仅清单 |
